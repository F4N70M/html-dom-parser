## **HtmlDomParser**

---

## 1. Введение

**HtmlDomParser** — это библиотека для парсинга HTML, которая преобразует HTML-строку в структурированное объектное дерево. В отличие от стандартного `DOMDocument`, она предоставляет расширенные возможности: автоматическое схлопывание inline-контента, семантическое извлечение данных (`data`), систему событий для расширения функциональности, гибкую обработку ошибок и модульную архитектуру.

Библиотека предназначена для:
- Извлечения и анализа содержимого HTML-документов.
- Построения удобного для навигации объектного представления.
- Интеграции в проекты, где требуется тонкая настройка парсинга.
- Создания расширений (модулей) для специфических задач.

Основные возможности:
- **Рекурсивный обход** с созданием контекста для каждого узла.
- **DataResolver** — определение основного содержимого элемента (URL, текст, значение атрибута).
- **InlineCollapser** — объединение фразовых элементов в единый текст с сохранением сущностей.
- **EventDispatcher** — подписка на события обработки узлов.
- **ErrorHandler** — централизованный сбор ошибок с уровнями серьезности.
- **ModuleManager** — подключение модулей через Composer.

Входные данные: HTML-строка. Выходные: объект `Document`, содержащий иерархию узлов (`Node`, `Element`, `TextElement`, и т.д.) с полями `data`, `label`, `entities` и другими метаданными.

## 2. Установка и подключение

### Требования
- PHP 8.1 или выше
- Расширение `dom` (встроенное)
- Composer (для управления зависимостями, даже при локальной установке)

### Варианты установки (локальная разработка)

#### Вариант 1: через локальный репозиторий в composer.json
Если библиотека находится в локальной директории (например, `packages/html-dom-parser`), добавьте в `composer.json` вашего проекта:

```json
{
    "repositories": [
        {
            "type": "path",
            "url": "packages/html-dom-parser"
        }
    ],
    "require": {
        "html-dom-parser/core": "*"
    }
}
```

Затем выполните `composer update`.

#### Вариант 2: ручное копирование
Скопируйте содержимое библиотеки в директорию вашего проекта (например, `lib/HtmlDomParser`) и настройте автозагрузку вручную:

```php
spl_autoload_register(function ($class) {
    $prefix = 'HtmlDomParser\\';
    $base_dir = __DIR__ . '/lib/HtmlDomParser/';
    $len = strlen($prefix);
    if (strncmp($prefix, $class, $len) !== 0) return;
    $relative_class = substr($class, $len);
    $file = $base_dir . str_replace('\\', '/', $relative_class) . '.php';
    if (file_exists($file)) require $file;
});
```

Рекомендуется использовать Composer даже для локальной установки, так как он автоматически обрабатывает автозагрузку PSR-4.

### Подключение автозагрузки Composer
Если вы используете Composer (рекомендуется), достаточно подключить:

```php
require_once 'vendor/autoload.php';
```

### Простейший пример использования
```php
use HtmlDomParser\Core\Parser;

$html = '<p>Пример <strong>текста</strong>.</p>';
$parser = new Parser($html);
$document = $parser->parse();

print_r($document->toArray()); // экспорт в массив
```

### Примечание о публикации
В текущей версии библиотека распространяется как локальный пакет. В будущем планируется публикация в репозитории Packagist для установки через `composer require`.

## 3. Файловая структура

Исходный код библиотеки организован в пространстве имён `HtmlDomParser` и расположен в директории `src/HtmlDomParser/`. Все интерфейсы вынесены в пространство `HtmlDomParser\Contract` для обеспечения слабой связанности и возможности подмены реализаций.

```
src/
└── HtmlDomParser/
    ├── Core/                                      # Ядро библиотеки
    │   ├── Node/                                  # Классы узлов дерева
    │   │   ├── Node.php                            # Базовый узел
    │   │   ├── Document.php                         # Корневой узел документа
    │   │   ├── Element.php                           # Базовый элемент
    │   │   ├── VoidElement.php                        # Самозакрывающийся элемент
    │   │   ├── TextElement.php                         # Текстовый узел
    │   │   ├── CommentElement.php                       # Комментарий
    │   │   └── ErrorElement.php                          # Узел-ошибка
    │   ├── Collection/                               # Коллекции
    │   │   └── ElementList.php                         # Коллекция элементов
    │   ├── Context/                                  # Контекстная обработка
    │   │   ├── NodeContext.php                         # Состояние узла при обходе
    │   │   └── NodeContextResolver.php                  # Создание контекста по правилам
    │   ├── Event/                                    # Событийная система
    │   │   └── EventDispatcher.php                      # Диспетчер событий
    │   ├── DataResolver.php                           # Определяет data для элементов
    │   ├── InlineCollapser.php                         # Сервис схлопывания inline-узлов
    │   ├── ErrorHandler.php                            # Централизованный сбор ошибок
    │   ├── ModuleManager.php                           # Управление модулями
    │   └── Parser.php                                  # Главный парсер
    │
    ├── Contract/                                   # Интерфейсы (все контракты)
    │   ├── ModuleInterface.php
    │   ├── EventDispatcherInterface.php
    │   ├── NodeContextInterface.php
    │   ├── NodeContextResolverInterface.php
    │   ├── InlineCollapserInterface.php
    │   ├── ElementListInterface.php
    │   └── ... (другие интерфейсы)
    │
    ├── Exception/                                  # Исключения
    │   ├── HtmlDomParserException.php
    │   ├── HtmlDomParserLoadException.php
    │   └── ...
    │
    ├── Config/                                     # Конфигурационные файлы
    │   └── TagRules.php                              # Правила обработки тегов
    │
    └── Module/                                      # Встроенные модули (опционально)
        ├── CssSelector/
        │   └── CssSelectorModule.php
        ├── XPath/
        │   └── XPathModule.php
        └── ...
```

### Ключевые особенности структуры

- **`Core/Node`** – все классы, представляющие узлы HTML-дерева. Они наследуют базовый класс `Node` и образуют иерархию для разных типов узлов.
- **`Core/Collection`** – содержит класс `ElementList`, реализующий `IteratorAggregate` и `Countable`, используется для хранения дочерних элементов.
- **`Core/Context`** – классы, управляющие состоянием узла в процессе рекурсивного обхода (`NodeContext`) и создающие контекст на основе правил (`NodeContextResolver`).
- **`Core/Event`** – реализация диспетчера событий. Сам диспетчер находится здесь, а его интерфейс – в `Contract`.
- **`Contract`** – все интерфейсы библиотеки. Это обеспечивает возможность замены реализаций (например, для тестирования) и чёткое разделение контрактов и реализаций.
- **`Config`** – файлы с настройками, которые могут быть переопределены пользователем (например, правила тегов).
- **`Module`** – директория для встроенных модулей. Каждый модуль может быть оформлен как отдельный пакет, но для разработки может временно располагаться здесь.

Такая организация позволяет легко ориентироваться в коде, поддерживать расширяемость и соответствует принципам PSR-4.

## 4. Архитектура

Библиотека построена вокруг рекурсивного обхода HTML-дерева с разделением ответственности между компонентами. Основная идея — преобразовать сырой `DOMDocument` в объектную модель (`Document`), обогащённую семантическими данными (`data`, `label`, `entities`), при этом предоставляя возможности расширения через события и модули.

### 4.1 Общая схема работы

```
HTML строка
    ↓
\DOMDocument (libxml)
    ↓
Рекурсивный обход (Parser)
    ├── для каждого узла создаётся NodeContext (содержит состояние и правила)
    ├── на ключевых этапах вызываются события (EventDispatcher)
    ├── после обработки детей выполняется схлопывание inline-контента (InlineCollapser)
    └── в конце формируется Element
    ↓
Document (корневой объект)
```

### 4.2 Ключевые компоненты

- **Parser** – главный класс, инициирующий процесс. Загружает HTML, создаёт корневой контекст и запускает рекурсивную обработку.
- **NodeContext** – объект, сопровождающий узел на всём пути обработки. Хранит:
  - оригинальный `\DOMNode`
  - ссылку на родительский контекст
  - дочерние элементы (после обработки)
  - правила включения (`includeType`, `childrenType`, `allowedTags`)
  - флаг `isVoid` (может ли узел иметь детей)
  - разрешённые теги для проверки дочерних узлов
- **NodeContextResolver** – создаёт контекст для узла, применяя правила из `TagRules.php` и наследование от родителя.
- **DataResolver** – определяет основное содержимое узла (`data`) на основе его имени и атрибутов (например, для `<a>` – атрибут `href`, для `<script>` – внутренний текст). Возвращает строку или `null`.
- **InlineCollapser** – объединяет последовательность inline-узлов (фразовых элементов и текст) в один элемент с единым текстом и массивом сущностей форматирования (`entities`). Используется для сжатия мелких узлов в читаемый текст.
- **EventDispatcher** – позволяет модулям подписываться на события (`pre-node`, `post-node`, `pre-inline-collapse`, `post-inline-collapse`) и модифицировать контекст. Все обработчики получают текущий `NodeContext` и возвращают его (возможно, изменённый).
- **ErrorHandler** – собирает ошибки, возникающие в процессе парсинга, с уровнями `NOTICE`, `WARNING`, `ERROR`. Может быть настроен на генерацию исключений при определённых уровнях.
- **ModuleManager** – управляет модулями, загружает их и передаёт им диспетчер событий для регистрации обработчиков.

### 4.3 Поток обработки узла (упрощённо)

1. Для текущего DOM-узла создаётся `NodeContext` через `NodeContextResolver`.
2. Вызывается событие `pre-node` (модули могут изменить контекст).
3. Если узел не void, рекурсивно обрабатываются все его дочерние узлы, для каждого создаётся свой контекст и элемент.
4. Полученные дочерние элементы собираются в `ElementList` и сохраняются в контексте.
5. Если все дочерние элементы являются inline (проверка через `allChildrenIsInline`), запускается схлопывание:
   - событие `pre-inline-collapse`
   - вызов `InlineCollapser::collapse()`, который возвращает новый `ElementList` (обычно с одним элементом)
   - событие `post-inline-collapse`
6. Вызывается событие `post-node`.
7. Контекст преобразуется в объект `Element` (или его наследника) через метод `toElement()`, который использует сохранённые данные (`data` из `DataResolver`, атрибуты, детей и т.д.).
8. Элемент возвращается на уровень выше.

### 4.4 Расширяемость

- **События** – позволяют внедрять дополнительную логику на любом этапе обработки узла, не изменяя ядро.
- **Модули** – регистрируют обработчики событий и могут добавлять новые методы в `Document` или `Element` (через динамическое расширение или композицию). Модули устанавливаются через Composer и автоматически подключаются.

### 4.5 Обработка ошибок

Ошибки классифицируются по трём уровням:
- `NOTICE` – незначительные проблемы (отсутствие правила для тега, пропущенный атрибут) – парсинг продолжается.
- `WARNING` – проблемы, влияющие на структуру (запрещённый дочерний элемент) – парсинг продолжается.
- `ERROR` – фатальные ошибки (невозможно загрузить HTML, превышение глубины рекурсии) – по умолчанию выбрасывается исключение, но поведение можно настроить.

Каждая ошибка представлена объектом `ErrorElement`, который замещает проблемный узел в дереве и содержит диагностическую информацию. Все ошибки собираются в `ErrorHandler` и доступны для анализа после парсинга.

## 5. Основные классы и интерфейсы

В этом разделе перечислены ключевые классы и интерфейсы библиотеки с кратким описанием их назначения. Подробная документация по каждому компоненту приведена в соответствующих отдельных документах (ссылки даны в разделе «Список документации»).

### 5.1 Классы ядра (Core)

| Класс | Назначение |
|-------|------------|
| `HtmlDomParser\Core\Parser` | Главный класс парсера. Принимает HTML, управляет процессом рекурсивного обхода и возвращает объект `Document`. |
| `HtmlDomParser\Core\DataResolver` | Извлекает основное содержимое (`data`) из DOM-узла согласно внутренней таблице правил (атрибуты или текст). |
| `HtmlDomParser\Core\InlineCollapser` | Объединяет последовательность inline-узлов в единый текст с массивом сущностей форматирования. Возвращает `ElementList`. |
| `HtmlDomParser\Core\ErrorHandler` | Централизованно собирает ошибки парсинга с уровнями `NOTICE`, `WARNING`, `ERROR`. Позволяет настраивать поведение (бросать исключение или продолжать). |
| `HtmlDomParser\Core\ModuleManager` | Обнаруживает, загружает и инициализирует модули, переданные через Composer или расположенные в директории `Module/`. |
| `HtmlDomParser\Core\Event\EventDispatcher` | Реализует событийную систему: хранит обработчики, вызывает их при наступлении события, передавая текущий контекст узла. |

### 5.2 Классы контекста и коллекций

| Класс | Назначение |
|-------|------------|
| `HtmlDomParser\Core\Context\NodeContext` | Сопровождает узел в процессе рекурсивного обхода. Хранит оригинальный `DOMNode`, родительский контекст, дочерние элементы, правила тега и флаги. |
| `HtmlDomParser\Core\Context\NodeContextResolver` | Создаёт контекст для DOM-узла на основе правил тегов и наследования от родительского контекста. |
| `HtmlDomParser\Core\Collection\ElementList` | Коллекция объектов `Element`, реализующая `IteratorAggregate` и `Countable`. Используется для хранения дочерних узлов в `Element` и контексте. |

### 5.3 Классы узлов (Node hierarchy)

Все классы узлов находятся в пространстве имён `HtmlDomParser\Core\Node`.

| Класс | Наследует | Назначение |
|-------|-----------|------------|
| `HtmlDomParser\Core\Node\Node` | – | Базовый класс для всех узлов. Содержит общие свойства `name`, `attributes` и методы для работы с ними. |
| `HtmlDomParser\Core\Node\Element` | `Node` | Представляет HTML-элемент. Содержит поля `data`, `label`, `entities`, флаги `isRawText`, `isEscapable`, а также методы для схлопывания. |
| `HtmlDomParser\Core\Node\VoidElement` | `Element` | Самозакрывающийся элемент (например, `<img>`, `<br>`). Не может содержать детей и не участвует в схлопывании. |
| `HtmlDomParser\Core\Node\TextElement` | `Element` | Текстовый узел. Всегда inline, `data` содержит текст после нормализации. |
| `HtmlDomParser\Core\Node\CommentElement` | `Element` | Узел-комментарий. Сохраняется только при `$keepComments = true`. |
| `HtmlDomParser\Core\Node\ErrorElement` | `Element` | Специализированный узел, замещающий проблемный HTML-узел. Содержит информацию об ошибке (уровень, тип, backtrace). |
| `HtmlDomParser\Core\Node\Document` | `Node` | Корневой узел документа. Содержит ссылку на корневой элемент, а также на обработчик ошибок (ErrorHandler) для доступа к ошибкам после парсинга. Методы: getRootElement(), getErrorHandler(), toArray(), toHtml(). |

### 5.4 Интерфейсы (Contract)

Все интерфейсы находятся в пространстве `HtmlDomParser\Contract` и определяют контракты для основных компонентов, что позволяет легко подменять реализации (например, для тестирования).

#### `HtmlDomParser\Contract\ModuleInterface`
Контракт для всех модулей; обеспечивает идентификацию, проверку зависимостей и инициализацию.
**Методы:**
- `getName(): string`
- `getDependencies(): array`
- `supportsCoreVersion(string $version): bool`
- `initialize(EventDispatcherInterface $dispatcher): void`

#### `HtmlDomParser\Contract\EventDispatcherInterface`
Определяет работу событийной системы: подписка и вызов обработчиков.
**Методы:**
- `subscribe(string $event, callable $handler, int $priority = 0): void`
- `dispatch(string $event, NodeContext $context): NodeContext`
- `hasListeners(string $event): bool`
- `clearListeners(string $event): void`

#### `HtmlDomParser\Contract\NodeContextInterface`
Контракт для контекста узла, предоставляющий состояние и методы управления.
**Методы:**
- `getNode(): \DOMNode`
- `getParent(): ?NodeContextInterface`
- `setParent(?NodeContextInterface $parent): void`
- `getChildren(): ElementListInterface`
- `setChildren(ElementListInterface $children): void`
- `getData(): mixed`
- `setData($data): void`
- `isVoid(): bool`
- `isInline(): bool`
- `isRawText(): bool`
- `isEscapable(): bool`
- `getAllowedTags(): array`
- `allChildrenIsInline(): bool`
- `toElement(): Element`

#### `HtmlDomParser\Contract\NodeContextResolverInterface`
Создаёт контекст для DOM-узла на основе правил и родительского контекста.
**Методы:**
`resolve(\DOMNode $node, ?NodeContextInterface $parentContext = null): NodeContextInterface`

#### `HtmlDomParser\Contract\InlineCollapserInterface`
Сервис схлопывания inline-узлов.
**Методы:**
- `collapse(ElementListInterface $children, array $options = []): ElementListInterface`
- `setParentContext(?NodeContextInterface $context): void`

#### `HtmlDomParser\Contract\ElementListInterface`
Коллекция элементов с базовыми операциями.
**Методы:**
- `push(Element $element): void`
- `get(int $index): ?Element`
- `count(): int`
- `getIterator(): \ArrayIterator`
- `toArray(): array`
- `filter(callable $callback): ElementListInterface`
- `map(callable $callback): array`

#### `HtmlDomParser\Contract\DataResolverInterface`
Извлекает основное содержимое узла (`data`). Другие методы не являются частью публичного API.
**Методы:**
- `resolve(\DOMNode $node): ?string`

Каждый из перечисленных компонентов подробно описан в отдельном документе, ссылки на которые приведены в конце настоящего руководства.

## 6. Работа с данными

Каждый узел HTML-дерева после парсинга представлен объектом-наследником `Node`. Для элементов (`Element` и его потомков) предусмотрены поля:
- **`data`** – основное содержимое элемента (URL, текст, значение атрибута), извлекаемое классом `DataResolver`.
- **`label`** – текстовая метка для элементов, где `data` содержит нетекстовую информацию (например, для ссылок).
- **`entities`** – массив сущностей форматирования, формируемый при схлопывании inline-контента.

Детальное описание правил извлечения `data`, формата сущностей и методов доступа приведено в отдельном документе **«DataResolver»**.

## 7. Конфигурация

Параметры парсинга задаются при создании объекта `Parser`. Дополнительная настройка поведения (например, обработка ошибок) выполняется через соответствующие компоненты до вызова метода `parse()`.

### 7.1 Параметры конструктора `Parser`

| Параметр |   Тип    | По умолчанию | Описание       |
|---|---|---|--------------------|
| `$html` | `string` | – | Исходная HTML-строка для парсинга. Обязательный. |
| `$keepComments` | `bool` | `false` | Если `true`, комментарии (`<!-- ... -->`) сохраняются в дереве как узлы `CommentElement`. Иначе игнорируются. |

Пример:
```php
$parser = new Parser($html, true); // сохранять комментарии
```

### 7.2 Конфигурация через другие компоненты

Некоторые настройки доступны через публичные методы компонентов, которые можно получить из парсера:

- **Обработка ошибок** – `$parser->getErrorHandler()` позволяет настроить уровни серьезности и поведение при ошибках. Подробнее см. раздел «Обработка ошибок».
- **Модули** – после инициализации модули могут предоставлять собственные методы конфигурации. Доступ к модулям осуществляется через `$parser->modules()`.

### 7.3 Примечание

Библиотека не предусматривает прямого переопределения встроенных правил обработки тегов или источников данных `DataResolver`. При необходимости изменения логики рекомендуется использовать события или создавать модули, которые модифицируют контекст на соответствующих этапах парсинга.

## 8. Обработка ошибок

В процессе парсинга могут возникать различные проблемы: от отсутствия правил для тега до фатальных сбоев загрузки HTML. Для управления этими ситуациями предусмотрена двухуровневая система:

- **`ErrorHandler`** – централизованный сборщик ошибок. Хранит все возникшие ошибки, позволяет настраивать поведение (выбрасывать исключение или продолжать работу) для каждого уровня серьезности.
- **`ErrorElement`** – специальный узел, наследующий `Element`, который замещает проблемный HTML-узел в дереве. Содержит диагностическую информацию: уровень ошибки, тип, человекочитаемое описание и backtrace.

### Уровни серьезности

- `NOTICE` – незначительные проблемы (отсутствие правила для тега, пропущенный атрибут). Парсинг продолжается.
- `WARNING` – проблемы, влияющие на интерпретацию (запрещённый дочерний элемент, ошибка в callback). Парсинг продолжается.
- `ERROR` – фатальные ошибки (невозможно загрузить HTML, превышение глубины рекурсии). По умолчанию выбрасывается исключение `HtmlDomParserException`.

Доступ к обработчику ошибок осуществляется через метод `Parser::getErrorHandler()`. Настройка поведения выполняется до вызова `parse()`:

```php
$parser = new Parser($html);
$handler = $parser->getErrorHandler();
$handler->setThrowOnError(false);  // не прерываться при ERROR
```

После парсинга все ошибки можно проанализировать:

```php
if ($handler->hasErrors()) {
    foreach ($handler->getErrorsBySeverity('warning') as $warning) {
        echo $warning->getLabel();
    }
}
```

Подробное описание уровней, источников ошибок, методов `ErrorHandler` и структуры `ErrorElement` приведено в отдельном документе **«ErrorHandler + ErrorElement»**.

## 9. Расширение библиотеки (События и модульность)

Библиотека спроектирована с возможностью расширения без изменения ядра. Для этого предусмотрены два взаимосвязанных механизма: **события** и **модули**. События позволяют внедрять пользовательскую логику в ключевые моменты обработки узлов, а модули — группировать такую логику в переиспользуемые пакеты.

## 9. Расширение библиотеки

Библиотека предоставляет два механизма для расширения функциональности без изменения ядра: **события** и **модули**.

### 9.1 События

В процессе рекурсивного обхода для каждого узла генерируются события. Модули могут подписываться на них и модифицировать контекст узла (`NodeContext`), влияя на дальнейшую обработку. Список событий:

- `pre-node` – после создания контекста, до обработки детей;
- `post-node` – после обработки детей и схлопывания, перед созданием элемента;
- `pre-inline-collapse` – перед схлопыванием inline-узлов;
- `post-inline-collapse` – после схлопывания inline-узлов.

Управление событиями осуществляется через `EventDispatcher` (подробнее в документации **«EventDispatcher»**).

### 9.2 Модули

Модули представляют собой законченные расширения (например, поддержка CSS-селекторов, XPath). Они реализуют интерфейс `ModuleInterface`, загружаются через `ModuleManager` и могут подписываться на события. 

**Процесс загрузки модулей:**  
При создании парсера `ModuleManager` автоматически сканирует секцию `extra.modules` в `composer.json` проекта, создаёт экземпляры указанных классов (с использованием автозагрузки PSR-4) и инициализирует их, передавая диспетчер событий. Модули могут подписываться на события в методе `initialize()`, а также предоставлять собственные публичные методы для внешнего вызова.

После инициализации модули доступны через метод `$parser->modules()->getModule('name')`. Это позволяет вызывать их публичные методы напрямую, независимо от событийной системы. Например, модуль CSS-селектора может предоставлять метод `querySelectorAll()`.

Детальное описание модульной системы приведено в отдельном документе **«Модульная система»**.

## 10. Производительность

Библиотека спроектирована с учётом эффективной обработки HTML-документов различного размера. Основные характеристики:

- **Рекурсивный обход** – каждый узел обрабатывается один раз, сложность O(N), где N – общее количество узлов.
- **Схлопывание inline-контента** – выполняется за линейное время относительно числа inline-узлов.
- **DataResolver** – работает за константное время для каждого узла.
- **Ленивое резолвление** – данные извлекаются только при первом обращении, что снижает накладные расходы.
- **Кэширование** – результаты повторных обращений к контексту не пересчитываются.
- **Единые экземпляры сервисов** – `InlineCollapser`, `DataResolver` и другие создаются один раз и переиспользуются.
