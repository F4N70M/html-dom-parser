# Введение в HtmlDomParser {#introduction}

**HtmlDomParser** — это PHP-библиотека для продвинутого парсинга HTML, которая преобразует исходный код в удобное объектное дерево с богатой семантической информацией. Это не просто очередной парсер, а полноценный фреймворк для построения семантически-обогащенного DOM-дерева.

## Что делает библиотеку особенной? {#what-makes-it-special}

В отличие от стандартных парсеров, которые возвращают сырое DOM-дерево, HtmlDomParser выполняет рекурсивный обход с **контекстной обработкой** каждого узла:

```
HTML-строка 
    → Загрузка в DOMDocument 
    → Рекурсивный обход с созданием контекста для каждого узла 
    → Обогащение семантической информацией 
    → Итоговое дерево (Document с элементами)
```

## Ключевые возможности {#key-features}

### 🧠 Контекстная обработка узлов {#contextual-processing}

Для каждого DOM-узла создается специальный объект **контекста**, который хранит:
- Ссылку на оригинальный узел
- Правила обработки тега (строчный, блочный, пустой)
- Информацию о разрешенных дочерних тегах
- Результаты промежуточных вычислений

Правила берутся из встроенной **карты контекстов тегов**, содержащей информацию для всех стандартных HTML-тегов.

Подробнее о контекстной обработке читайте в разделе [Система контекстов](./02-core--02-context-system.md).

### 🔗 Схлопывание строчного контента {#inline-collapsing}

Одна из ключевых особенностей — объединение последовательности строчных элементов (текст, выделения, ссылки) в один элемент с единым текстом и массивом **сущностей форматирования**:

```php
// HTML: <p>Это <b>жирный</b> и <i>курсивный</i> текст</p>

// После парсинга:
$element->getLabel();
// "Это жирный и курсивный текст"

$element->getEntities()->toArray(); 
// [
//   ['type' => 'b', 'start' => 4, 'end' => 10],
//   ['type' => 'i', 'start' => 13, 'end' => 22]
// ]
```

Детальное описание механизма схлопывания — в разделе [InlineCollapser](./02-core--03-utilities.md#inline-collapser), а работа с сущностями форматирования — в разделе [Модель данных элемента](./02-core--01-data-model.md#entities).

### 🧩 Расширяемость через события и модули {#extensibility}

- **Событийная модель** — в ключевых точках обработки генерируются события. Любой желающий может подписаться на них и внедрить свою логику.
- **Модульная архитектура** — модули оформляются как Composer-пакеты, автоматически обнаруживаются и могут предоставлять свой API для работы с готовым деревом.

### ⚠️ Продвинутая обработка ошибок {#error-handling}

Все ошибки парсинга централизованно собираются в обработчике ошибок и классифицируются по уровням серьезности:
- **Уведомления (notice)** — не влияют на результат
- **Предупреждения (warning)** — возможны незначительные проблемы
- **Фатальные ошибки (error)** — результат может быть некорректен

Для каждой проблемы создается специальный **узел-ошибка**, замещающий проблемный HTML-узел в дереве. Поведение для каждого уровня можно настроить.

Читайте подробнее в разделе [Обработка ошибок](./02-core--04-error-handling.md).

## Итоговая модель данных {#resulting-data-model}

Результатом работы является дерево, состоящее из объектов, реализующих систему интерфейсов (полный список см. в [Справочнике API](./04-appendix--02-api-reference.md)):

- **[`DocumentInterface`](./04-appendix--02-api-reference.md#documentinterface)** — корневой объект, содержащий всё дерево.
- **[`ElementInterface`](./04-appendix--02-api-reference.md#elementinterface)** — основной тип узла для тегов, содержит:
  - **Data** — основное смысловое содержимое.
  - **Label** — текстовая метка.
  - **[`EntityListInterface`](./04-appendix--02-api-reference.md#entitylistinterface)** — коллекция сущностей форматирования.
  - **[`ElementListInterface`](./04-appendix--02-api-reference.md#elementlistinterface)** — коллекция дочерних элементов.
- **[`NodeInterface`](./04-appendix--02-api-reference.md#nodeinterface)** — базовая единица с именем и атрибутами.
